## 2. 逐项详细说明

### R-026 | P0 | `econ.set_mining` 工具缺失

**来源**：第2轮审查 `sc2_agent/tools/econ.py`

**设计文档要求**（FR-05）：`econ.set_mining — (base_id, gas_count?) 调整某基地气矿工人数`

**当前状态**：未实现。econ 命名空间只有 4 个工具（`gather`、`transfer_workers`、`expand`、`build_gas`），缺少第 5 个。

**影响**：Agent 无法精细控制气矿工人分配。

**整改**：新建 `EconSetMiningTool`，参数 `base_id: int` + `gas_count: int | None`。

---

### R-027 | P0 | `squad.list` 工具缺失

**来源**：第2轮审查 `sc2_agent/tools/squad.py`

**设计文档要求**（FR-05）：`squad.list — 当前所有小队的摘要`

**当前状态**：未实现。

**影响**：Agent 创建小队后无法列出它们。

**整改**：新建 `SquadListTool`，返回所有小队名 + 成员数 + 成员 tag 列表。

---

### R-028 | P1 | `timer.monitor` metric 三处不一致

**来源**：第2轮 + 第3轮审查

**涉及文件**：
| 文件 | 状态 |
|------|------|
| `timer/models.py` `MonitorMetric` 类型 | 含9个（有 unit_distance/unit_in_region） |
| `tools/timer.py` JSON Schema enum | 仅7个，缺两个 |
| `timer/scheduler.py` `_monitor_value()` | 仅7个实现，两个返回 None |

**影响**：Agent 无法设置"单位到达区域时唤醒"或"单位接近目标时唤醒"。

**整改**：三处同步补齐 `unit_distance` 和 `unit_in_region`。

---

### R-029 | P1 | `obs.resources` 缺少收入字段

**来源**：第2轮审查

**当前返回**：`{game_time, minerals, gas, supply_used, supply_cap, supply_available}`——缺少 `income_min` 和 `income_gas`。

**影响**：Agent 无法获取当前实际收入速率，plan.simulate 依赖 Agent 手动估算。

**整改**：`ObsResourcesTool.execute()` 增加 `income_min` 和 `income_gas` 字段。

---

### R-030 | P1 | `review.params` 不验证 tag 真实存在

**来源**：第2轮审查

**根因**：`ReviewParamsTool._has_producer_tag()` 只检查 key 存在，不调 `bot.find_by_tag()` 验证。

**影响**：假阳性——tag 格式正确但实际不存在 → PASS。

**整改**：注入 ObservationProvider，验证 tag 存在性。

### R-036 | P0 | timer 命令执行失败不唤醒 Agent

**来源**：5.30 执行模型分析

**场景**：Agent 排期 `timer.command(SCV tag=42, build.structure)` → SCV 在 t=125 死亡 → t=127 执行 → TAG_NOT_FOUND → 命令标记 "failed" → Agent 继续休眠 ❌

**涉及层级**：
| 层级 | 处理 |
|------|------|
| `cmd.py` `_resolve_units()` | 返回 TAG_NOT_FOUND + ok=False ✅ |
| `scheduler.py` `tick()` | 标记 status="failed"，写 run_history ✅ |
| `bot.py` `_tick_scheduler()` | 只打 debug 日志 ❌ |
| `SchedulerResult` | 不区分成功/失败 ❌ |

**整改**：`SchedulerResult` 加 `failed` 字段 → `tick()` 收集 → `_tick_scheduler()` 检测到非空 → `wake_to_thinking()`。

---

### R-037 | P0 | SquadCreateTool 与其他 squad 工具 dict 隔离

**来源**：5.30 并行审查

**根因**：`bot.py` 中 `squads = {}` 传入 add/remove/disband/order，但 `SquadCreateTool.__init__` 自建 `self._squads = {}`。

**影响**：squad.create 写入内部 dict，squad.add 在空的局部 dict 中查找 → SQUAD_NOT_FOUND。所有 squad 操作不可用。

**整改**：共享同一个 dict 传入所有 squad 工具。

---

### R-038 | P1 | cmd.py 18 个工具动作执行阶段无 try/except

**来源**：5.30 并行审查

**根因**：`_resolve_units()` 通过后，`unit.move()` 等调用无 try/except。单位中途死亡 → 异常穿透。

**整改**：18 个 cmd 工具每处包裹 `try/except Exception as e`，返回结构化错误码（参考 `BuildStructureTool` 的 BUILD_ERROR 模式）。

---

### R-039 | P1 | BuildTrainTool 部分成功返回 ok=True 不告知期望 count

**来源**：5.30 并行审查

**根因**：`for _ in range(count)` 中 `except Exception: break` 吞异常，`ok=True, trained=2` 但 Agent 期望 5。

**整改**：返回值加 `requested: count` 字段，异常时记录到 errors。

---

### R-040 | P1 | TimerRunRecord 丢弃 Result.code / Result.meta

**来源**：5.30 并行审查

**根因**：`scheduler.py:59` 只取 `result.error`（字符串），丢弃 `.code` 和 `.meta`。

**整改**：`TimerRunRecord` 加 `error_code` 和 `error_meta` 字段。

---

### R-041 | P1 | SchedulerResult 不区分成功/失败

**来源**：5.30 并行审查（R-036 姊妹问题）

**根因**：`executed` 是平面列表，bot 只打日志。

**整改**：`SchedulerResult` 加 `failed` 字段；bot 检测失败自动唤醒。

### R-042 | P1 | review.logic SubAgent 失败伪装成 success+WARN

**来源**：5.30 并行审查

**根因**：SubAgent 调用失败 → `LogicReview(verdict="WARN", issues=[...])` → ToolRegistry 包装为 `Result.success`。Agent 看到 ok=True 无法区分。

**整改**：SubAgent 失败时直接返回 `Result.failure`。

### R-043 | P1 | econ.build_gas 无可用 worker 时静默 ok=True

**来源**：5.30 并行审查

**根因**：`econ.py:151-158` 两个 fallback 都不满足时直接 `return {"ok": True}`。

**整改**：都失败时返回 `{"ok": False, "code": "NO_WORKER_AVAILABLE"}`。

### R-044 | P1 | econ 部分失败时 errors 被 scheduler 丢弃

**来源**：5.30 并行审查

**根因链路**：econ 返回 `ok=True, errors=[...]` → registry 存 errors 到 `data.errors` → scheduler 只取 `result.error`（None）→ run_history 显示成功。

**整改**：scheduler 读取 `data.errors` 写入 TimerRunRecord。

### R-050 | P1 | Agent 混淆 timer.monitor 和 timer.command 职责

**来源**：5.30 实测

**现象**：Agent 用 `timer.monitor(minerals>=100)` 替代 `timer.command(build.structure)` 做建造。

**根因**：system prompt 和 main-flow skill 未区分：
- `timer.command` = 在 at_time 时刻执行指定工具（命令执行器）
- `timer.monitor` = 条件满足时暂停游戏并唤醒 Agent（条件唤醒器）

**整改**：在 prompt 和 main-flow skill 中明确区分两者的职责边界。

### R-051 | P0 | WORKER_TYPE_NAMES 大小写不匹配

**来源**：5.30 实测 Session 分析

**根因**：`query.py:297` `WORKER_TYPE_NAMES = {"scv", "probe", "drone"}`（小写），但 `canonical_name("SCV")` 返回 `"SCV"`（大写）。`"SCV" in {"scv", ...}` → `False`。

**影响**：`query.find_workers` 永远返回空。

**整改**：改为 `{"SCV", "Probe", "Drone"}`。

### R-052 | P0 | BurnySC2ObservationAdapter 不读取中立单位

**来源**：5.30 实测 Session 分析

**根因**：`snapshot()` 只读 `bot.units` + `bot.structures`，不读 `bot.mineral_field` / `bot.vespene_geyser`。

**影响**：MineralField / VespeneGeyser 查询全空，`obs.unit(resource_tag)` 永远 TAG_NOT_FOUND。

**整改**：adapter 追加读取 neutral 资源单位；`ObsUnitTool` 增加 bot fallback。

---

### R-053 | P1 | plan.simulate 默认 income_rate=0

**来源**：5.30 实测 Session 分析

**根因**：`SimulationState.from_dict()` 用 `data.get("mineral_income_rate", 0.0)`，Agent 不传收入 → 模拟收入恒为0 → 全部报 INSUFFICIENT_RESOURCES。且 Agent 无途径获取收入率（obs.resources 不返回——R-029）。

**影响**：Agent 两次 simulate 均全败，放弃规划改用 monitor 兜底。

**整改**：新增 `plan.initial_state` 工具，内部调 `bot._make_simulation_state()` 自动计算 income_rate。

---

### R-054 | P1 | PAUSED_THINKING 中允许直接执行写操作

**来源**：5.30 实测 Session 分析

**根因**：Agent 直接调 `build.train(SCV)` 立即执行成功，不经 timer.command → staging → 审查。

**影响**：即时执行绕过整个 commit 流程，后续 simulate 需手动补齐。

**整改**：prompt 明确禁止 PAUSED_THINKING 中直接调写工具；ToolRegistry 可加状态校验。

---

### R-055 | P1 | review.plan 不检查模板目标完成度

**来源**：5.30 实测 Session 分析

**根因**：staging 无 SupplyDepot，但 `review.plan` 返回 PASS。审查 subAgent 未收到模板内容作上下文。

**整改**：review.plan 传入 build_order 模板内容，增加"模板覆盖度"审查维度。

---

### R-056 | P0 | 85 个工具 schema 全量发给 LLM ⭐实测发现

**来源**：2026-05-30 系统实测 + 日志分析

**根因**：`runner.py:92` 每次 LLM 调用都传 `spec.tools.schemas()`——全部 85 个工具的完整 JSON Schema。设计文档虽写了渐进式披露（3.6 节），但实际实现把所有 schema 一次性发给 LLM，导致：
- 请求体过大，API 可能因超出限制而报错
- Agent 连第一轮调用都没完成就 abort
- 85 个 schema 中绝大多数在当前迭代中完全用不到

**整改**：
- `ToolRegistry` 新增 `_active_namespaces`，初始只激活 `{"ctrl"}`（4 个工具）
- `schemas()` 只返回活跃命名空间的工具
- `ctrl.discover_tools(namespace)` 激活指定命名空间并返回其下工具 schema
- System prompt 中已有命名空间摘要（11 行），引导 Agent 按需调用 `ctrl.discover_tools`

**已完成**：ToolRegistry + ctrl.discover_tools 已改造，初始发送从 85 个 schema 降至 4 个。

### 5.5 实测中暴露的持续性系统问题

#### R-057 | P2 | Agent 高频 abort（LLM 不收敛）

**现象**：约 60% 的运行中，Agent 在 46-59 次工具调用后回空响应（`content: null`，无 `tool_calls`），AgentRunner 将之视为 `stop_reason="completed"` 并自动 `ctrl.abort`，导致决策周期未产生任何 timer 命令。

**根因**：DeepSeek v4-flash 模型在长对话（80k+ tokens）中偶发行为波动。成功时 Agent 在 34-43 次调用后 commit，失败时在 46-59 次调用后迷失方向，返回空响应。这是模型推理层面的不确定性，非代码 bug。

**已尝试的缓解措施**：

| 措施 | 效果 |
|------|------|
| 增加 max_iterations（20→40→60→80） | 无效——Agent 在达到上限前就回空 |
| prompt 加"时间线控制"约束 | 无效——LLM 不响应文本约束 |
| prompt 加 ⓪ 步预激活命名空间 | 改善——commit 率从 ~20% 提升到 ~60% |
| abort 后自动重试 | 无效——重试需游戏帧推进，abort 后游戏停滞 |
| abort 后注册 fallback_wake monitor 进入 RUNNING_SLEEP | 偶有成功——在阶段二中 3 次触发 |

**当前状态**：未完全解决。模型层面的不确定性无法在应用代码中根除。系统的 commit 率约 40%，依赖多周期恢复。后续可尝试换用更稳定的模型或减少上下文长度。

#### R-058 | P2 | burnysc2 观测断言崩溃（simulation_loop: 0）

**现象**：Agent 在第一个 `on_step` 中 abort 后，burnysc2 主循环尝试获取下一帧观测数据时触发：

```
File "sc2/client.py", line 145, in observation
    assert result.HasField("observation")
AssertionError: step { simulation_loop: 0 }
```

**根因**：非实时模式（`realtime=False`）下，burnysc2 期望 `on_step` 快速返回后 SC2 引擎立即产生下一帧观测。但 Agent 的 LLM 调用耗时 100+ 秒，远超 burnysc2 的预期窗口。当 `on_step` 返回时（无论 commit 还是 abort），SC2 引擎可能还处于中间状态，未准备好下发观测数据。

**为何 commit 后不一定崩**：commit 后 TimerStore 中有命令，游戏执行它们的过程中时钟自然推进了若干帧，burnysc2 有机会产生有效观测。abort 后 TimerStore 为空，游戏时钟无推进，burnysc2 瞬间请求下一帧观测但引擎未响应。

**已实施的容错**：

| 位置 | 措施 |
|------|------|
| `_save_snapshot_and_events()` | 改用 `_obs_store.snapshot()`（缓存快照），不再调 adapter 的实时 `snapshot()` |
| `_tick_scheduler()` | 观测获取外套 `try/except`，异常时静默跳过该 tick |
| `main.py` | `AssertionError` 捕获后记为 `"Game ended (normal)"` 并返回 exit code 0，不再视为崩溃 |

**当前状态**：代码层面已容错——abort 后游戏正常结束而非崩溃退出。但 abort 导致的无效运行仍需从模型层面改善（见 R-057）。

#### R-059 | P1 | plan.simulate 和 timer.command 命令格式不一致 ⭐实测发现

**现象**：Session 20-24-24 中，Agent 在 `plan.simulate` 里构造命令使用 `"tool"`/`"args"` 格式：
```json
{"tool": "build.train", "args": {"unit_type": "SCV", ...}, "item_name": "SCV", "at_time": 0.0}
```
而 `timer.command` 要求 `"tool_name"`/`"arguments"` 格式：
```json
{"tool_name": "build.train", "arguments": {"unit_type": "SCV", ...}, "at_time": 0.0}
```

**根因**：`simulator.py` 的 `PlanCommand.from_dict()` 只识别 `"arguments"` 子字典（timer.command 格式），不认识 `"args"`。Agent 在 simulate 里用 `"args"` 时，`data.get("arguments")` 返回 `None`，导致 `item_name` 只能从顶层的 `"item_name"` 字段取——但这个字段 Agent 必须手动添加。

**影响**：Agent 在 simulate 阶段反复调参（Session 中 9 次），但从未进入 timer.command 注册阶段。80 次迭代全耗在 simulate 上，零命令提交。

**修复**：`simulator.py:121-122` 增加对 `"args"` 子字典的识别：
```python
if not args:
    args = data.get("args", {}) if isinstance(data.get("args"), dict) else {}
```
Agent 现在可以用 `{"tool": "X", "args": {...}, "at_time": N}` 格式直接给 simulate 传命令，simulate 会正确解析 `args.unit_type`、`args.structure_tag`、`args.worker_tag` 等字段。

**已完成**：simulator.py 已修改，16 tests passed。

### 5.6 工具臃肿相关问题的汇集与分析

以下问题均由 Session 20-24-24 的实测日志暴露，根因均指向"工具系统过于臃肿"。

---

#### 问题 1：`ctrl.discover_tools` 返回全量 schema（每次 2000+ token）

Agent 在第一次迭代中并行调用了 10 个 `ctrl.discover_tools`，每个返回了对应命名空间下所有工具的完整 JSON Schema（参数名、类型、描述、enum 值、required 字段）。10 个响应的 tool message 合计约 **20,000+ token**，全被写入 session 上下文，且在后续 LLM 调用中持续携带。

以 obs 为例：Agent 只需要 `obs.resources`、`obs.units`、`obs.structures`、`obs.enemy_visible` 四个工具，但 discover 返回了全部 12 个 obs 工具的完整 schema——包括开局完全用不到的 `obs.scores`、`obs.upgrades`、`obs.enemy_inferred`。

**建议**：`ctrl.discover_tools` 增加 `tool_names: list[str]` 可选参数，允许按需只获取指定工具的 schema。

---

#### 问题 2：`plan.initial_state` 返回冗余数据

Session 第 37 行的 `plan.initial_state` 返回值中，`producer_available_at` 字典包含了 12 个 SCV 的全部 24 个条目：
```json
"producer_available_at": {
  "CommandCenter:4314365953": 0.0, "4314365953": 0.0,
  "SCV:4314628097": 0.0, "4314628097": 0.0,
  "SCV:4314890241": 0.0, "4314890241": 0.0,
  ...  // 共 24 个条目
}
```
SCV 不是生产者（不能训练单位），它们出现在 `producer_available_at` 中毫无意义。这些冗余条目膨胀了上下文、增加了 LLM 的认知负担，且 Agent 在每轮 simulate 中都要重复携带这些数据。

**建议**：`_make_simulation_state()` 中过滤掉非生产型 unit（SCV 不应出现在 `producer_available_at` 中）。

---

#### 问题 3：`obs.units` 返回全量单位细节

Agent 调用 `obs.units(alliance="self")` 时返回了 12 个 SCV 的完整数据——每个包含 tag、坐标、血量、orders（含 target tag）、status 等字段。Agent 实际只需要"12 个 SCV，都在采矿"这个结论。

**建议**：`obs.units` 考虑支持 `summary: bool` 参数，返回计数摘要而非全量细节。

---

#### 问题 4：Agent 激活了不必要的前期命名空间

Session 中 Agent 激活了全部 10 个命名空间，包括 `squad`（8 个工具）、`cmd`（18 个工具）、`econ`（5 个工具）——这些在开局前 60 秒完全用不到。但 `ctrl.discover_tools` 的设计鼓励了"一次性全激活"的行为。

**建议**：在 system prompt 的 ⓪ 步中增加引导——"开局只激活 obs、plan、build、timer。cmd/econ/squad 在需要时再激活。"

---

#### 问题 5：`plan.build_order` 被调用 3 次才拿到模板详情

Agent 调了 3 次 `plan.build_order` 才拿到模板内容——第一次无参（列出模板）、第二次用错误参数名 `opening`、第三次正确拿到 `terran_1rax_expand`。这是因为 `plan.build_order` 的 schema 中参数名叫 `name`（非 `opening`），Agent 猜错了。

**建议**：`plan.build_order` 无参时直接返回第一个（默认）模板的完整内容，而非仅列出模板名列表。

---

#### 问题 6：`plan.build_time` 返回 `producer_type` 不一致

`plan.build_time("Barracks")` 返回 `producer_type: "SCV"`——表示"由 SCV 建造"。但 `plan.build_time("SCV")` 返回 `producer_type: "CommandCenter"`——表示"由 CommandCenter 训练"。两者语义相同（"谁生产这个东西"），但 `build.train` 的参数叫 `structure_tag`（指 CommandCenter），`build.structure` 的参数叫 `worker_tag`（指 SCV）。参数名的不一致增加了 Agent 的理解成本。

---

#### 问题 7：`query.find_units` 的 type_name 过滤区分大小写

Agent 查 `VespeneGeyser` 返回空，因为 snapshot 中 type_name 被标准化为 `VESPENEGEYSER`。`canonical_name` 会保持原始输入的大小写（不在 ALIASES 中的名称原样返回），导致大小写不匹配。Agent 浪费了 2 轮迭代尝试不同拼写。

**建议**：`query.find_units` 和同类过滤工具在比较 type_name 时统一使用 `canonical_name()` 两边标准化后的小写比较。

---

#### 问题 8：`OrbitalCommand` 不在 costs.py 中

Agent 尝试查 `OrbitalCommand` 和 `Orbital` 的建造信息，均返回 UNKNOWN_ITEM。这是人族关键升级（CommandCenter → OrbitalCommand），缺失导致 Agent 无法正确规划科技路线。
