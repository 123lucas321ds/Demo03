## 2. 逐项详细说明

### R-026 | P0 | `econ.set_mining` 工具缺失

**来源**：第2轮审查 `sc2_agent/tools/econ.py`

**设计文档要求**（FR-05）：`econ.set_mining — (base_id, gas_count?) 调整某基地气矿工人数`

**当前状态**：未实现。econ 命名空间只有 4 个工具（`gather`、`transfer_workers`、`expand`、`build_gas`），缺少第 5 个。

**影响**：Agent 无法精细控制气矿工人分配。

**整改**：新建 `EconSetMiningTool`，参数 `base_id: int` + `gas_count: int | None`。

---

### R-027 | P0 | `squad.list` 工具缺失

**来源**：第2轮审查 `sc2_agent/tools/squad.py`

**设计文档要求**（FR-05）：`squad.list — 当前所有小队的摘要`

**当前状态**：未实现。

**影响**：Agent 创建小队后无法列出它们。

**整改**：新建 `SquadListTool`，返回所有小队名 + 成员数 + 成员 tag 列表。

---

### R-028 | P1 | `timer.monitor` metric 三处不一致

**来源**：第2轮 + 第3轮审查

**涉及文件**：
| 文件 | 状态 |
|------|------|
| `timer/models.py` `MonitorMetric` 类型 | 含9个（有 unit_distance/unit_in_region） |
| `tools/timer.py` JSON Schema enum | 仅7个，缺两个 |
| `timer/scheduler.py` `_monitor_value()` | 仅7个实现，两个返回 None |

**影响**：Agent 无法设置"单位到达区域时唤醒"或"单位接近目标时唤醒"。

**整改**：三处同步补齐 `unit_distance` 和 `unit_in_region`。

---

### R-029 | P1 | `obs.resources` 缺少收入字段

**来源**：第2轮审查

**当前返回**：`{game_time, minerals, gas, supply_used, supply_cap, supply_available}`——缺少 `income_min` 和 `income_gas`。

**影响**：Agent 无法获取当前实际收入速率，plan.simulate 依赖 Agent 手动估算。

**整改**：`ObsResourcesTool.execute()` 增加 `income_min` 和 `income_gas` 字段。

---

### R-030 | P1 | `review.params` 不验证 tag 真实存在

**来源**：第2轮审查

**根因**：`ReviewParamsTool._has_producer_tag()` 只检查 key 存在，不调 `bot.find_by_tag()` 验证。

**影响**：假阳性——tag 格式正确但实际不存在 → PASS。

**整改**：注入 ObservationProvider，验证 tag 存在性。

### R-036 | P0 | timer 命令执行失败不唤醒 Agent

**来源**：5.30 执行模型分析

**场景**：Agent 排期 `timer.command(SCV tag=42, build.structure)` → SCV 在 t=125 死亡 → t=127 执行 → TAG_NOT_FOUND → 命令标记 "failed" → Agent 继续休眠 ❌

**涉及层级**：
| 层级 | 处理 |
|------|------|
| `cmd.py` `_resolve_units()` | 返回 TAG_NOT_FOUND + ok=False ✅ |
| `scheduler.py` `tick()` | 标记 status="failed"，写 run_history ✅ |
| `bot.py` `_tick_scheduler()` | 只打 debug 日志 ❌ |
| `SchedulerResult` | 不区分成功/失败 ❌ |

**整改**：`SchedulerResult` 加 `failed` 字段 → `tick()` 收集 → `_tick_scheduler()` 检测到非空 → `wake_to_thinking()`。

---

### R-037 | P0 | SquadCreateTool 与其他 squad 工具 dict 隔离

**来源**：5.30 并行审查

**根因**：`bot.py` 中 `squads = {}` 传入 add/remove/disband/order，但 `SquadCreateTool.__init__` 自建 `self._squads = {}`。

**影响**：squad.create 写入内部 dict，squad.add 在空的局部 dict 中查找 → SQUAD_NOT_FOUND。所有 squad 操作不可用。

**整改**：共享同一个 dict 传入所有 squad 工具。

---

### R-038 | P1 | cmd.py 18 个工具动作执行阶段无 try/except

**来源**：5.30 并行审查

**根因**：`_resolve_units()` 通过后，`unit.move()` 等调用无 try/except。单位中途死亡 → 异常穿透。

**整改**：18 个 cmd 工具每处包裹 `try/except Exception as e`，返回结构化错误码（参考 `BuildStructureTool` 的 BUILD_ERROR 模式）。

---

### R-039 | P1 | BuildTrainTool 部分成功返回 ok=True 不告知期望 count

**来源**：5.30 并行审查

**根因**：`for _ in range(count)` 中 `except Exception: break` 吞异常，`ok=True, trained=2` 但 Agent 期望 5。

**整改**：返回值加 `requested: count` 字段，异常时记录到 errors。

---

### R-040 | P1 | TimerRunRecord 丢弃 Result.code / Result.meta

**来源**：5.30 并行审查

**根因**：`scheduler.py:59` 只取 `result.error`（字符串），丢弃 `.code` 和 `.meta`。

**整改**：`TimerRunRecord` 加 `error_code` 和 `error_meta` 字段。

---

### R-041 | P1 | SchedulerResult 不区分成功/失败

**来源**：5.30 并行审查（R-036 姊妹问题）

**根因**：`executed` 是平面列表，bot 只打日志。

**整改**：`SchedulerResult` 加 `failed` 字段；bot 检测失败自动唤醒。

---

### R-042 | P1 | review.logic SubAgent 失败伪装成 success+WARN

**来源**：5.30 并行审查

**根因**：SubAgent 调用失败 → `LogicReview(verdict="WARN", issues=[...])` → ToolRegistry 包装为 `Result.success`。Agent 看到 ok=True 无法区分。

**整改**：SubAgent 失败时直接返回 `Result.failure`。

---

### R-043 | P1 | econ.build_gas 无可用 worker 时静默 ok=True

**来源**：5.30 并行审查

**根因**：`econ.py:151-158` 两个 fallback 都不满足时直接 `return {"ok": True}`。

**整改**：都失败时返回 `{"ok": False, "code": "NO_WORKER_AVAILABLE"}`。

### R-044 | P1 | econ 部分失败时 errors 被 scheduler 丢弃

**来源**：5.30 并行审查

**根因链路**：econ 返回 `ok=True, errors=[...]` → registry 存 errors 到 `data.errors` → scheduler 只取 `result.error`（None）→ run_history 显示成功。

**整改**：scheduler 读取 `data.errors` 写入 TimerRunRecord。

### R-050 | P1 | Agent 混淆 timer.monitor 和 timer.command 职责

**来源**：5.30 实测

**现象**：Agent 用 `timer.monitor(minerals>=100)` 替代 `timer.command(build.structure)` 做建造。

**根因**：system prompt 和 main-flow skill 未区分：
- `timer.command` = 在 at_time 时刻执行指定工具（命令执行器）
- `timer.monitor` = 条件满足时暂停游戏并唤醒 Agent（条件唤醒器）

**整改**：在 prompt 和 main-flow skill 中明确区分两者的职责边界。

### R-051 | P0 | WORKER_TYPE_NAMES 大小写不匹配

**来源**：5.30 实测 Session 分析

**根因**：`query.py:297` `WORKER_TYPE_NAMES = {"scv", "probe", "drone"}`（小写），但 `canonical_name("SCV")` 返回 `"SCV"`（大写）。`"SCV" in {"scv", ...}` → `False`。

**影响**：`query.find_workers` 永远返回空。

**整改**：改为 `{"SCV", "Probe", "Drone"}`。

### R-052 | P0 | BurnySC2ObservationAdapter 不读取中立单位

**来源**：5.30 实测 Session 分析

**根因**：`snapshot()` 只读 `bot.units` + `bot.structures`，不读 `bot.mineral_field` / `bot.vespene_geyser`。

**影响**：MineralField / VespeneGeyser 查询全空，`obs.unit(resource_tag)` 永远 TAG_NOT_FOUND。

**整改**：adapter 追加读取 neutral 资源单位；`ObsUnitTool` 增加 bot fallback。

---

### R-053 | P1 | plan.simulate 默认 income_rate=0

**来源**：5.30 实测 Session 分析

**根因**：`SimulationState.from_dict()` 用 `data.get("mineral_income_rate", 0.0)`，Agent 不传收入 → 模拟收入恒为0 → 全部报 INSUFFICIENT_RESOURCES。且 Agent 无途径获取收入率（obs.resources 不返回——R-029）。

**影响**：Agent 两次 simulate 均全败，放弃规划改用 monitor 兜底。

**整改**：新增 `plan.initial_state` 工具，内部调 `bot._make_simulation_state()` 自动计算 income_rate。

---

### R-054 | P1 | PAUSED_THINKING 中允许直接执行写操作

**来源**：5.30 实测 Session 分析

**根因**：Agent 直接调 `build.train(SCV)` 立即执行成功，不经 timer.command → staging → 审查。

**影响**：即时执行绕过整个 commit 流程，后续 simulate 需手动补齐。

**整改**：prompt 明确禁止 PAUSED_THINKING 中直接调写工具；ToolRegistry 可加状态校验。

---

### R-055 | P1 | review.plan 不检查模板目标完成度

**来源**：5.30 实测 Session 分析

**根因**：staging 无 SupplyDepot，但 `review.plan` 返回 PASS。审查 subAgent 未收到模板内容作上下文。

**整改**：review.plan 传入 build_order 模板内容，增加"模板覆盖度"审查维度。

---

### R-056 | P0 | 85 个工具 schema 全量发给 LLM ⭐实测发现

**来源**：2026-05-30 系统实测 + 日志分析

**根因**：`runner.py:92` 每次 LLM 调用都传 `spec.tools.schemas()`——全部 85 个工具的完整 JSON Schema。设计文档虽写了渐进式披露（3.6 节），但实际实现把所有 schema 一次性发给 LLM，导致：
- 请求体过大，API 可能因超出限制而报错
- Agent 连第一轮调用都没完成就 abort
- 85 个 schema 中绝大多数在当前迭代中完全用不到

**整改**：
- `ToolRegistry` 新增 `_active_namespaces`，初始只激活 `{"ctrl"}`（4 个工具）
- `schemas()` 只返回活跃命名空间的工具
- `ctrl.discover_tools(namespace)` 激活指定命名空间并返回其下工具 schema
- System prompt 中已有命名空间摘要（11 行），引导 Agent 按需调用 `ctrl.discover_tools`

**已完成**：ToolRegistry + ctrl.discover_tools 已改造，初始发送从 85 个 schema 降至 4 个。
