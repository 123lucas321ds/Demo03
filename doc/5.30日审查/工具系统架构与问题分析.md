### 1.1 核心抽象

工具系统以三个类构成骨架：

- **`Tool`**（`sc2_agent/tools/base.py`）：抽象基类，定义工具的全部契约。每个工具拥有 `name`（字符串）、`description`、`parameters`（JSON Schema）、`read_only`（布尔标志）和 `execute()` 异步方法。
- **`ToolRegistry`**（`sc2_agent/tools/registry.py`）：中心注册表，持有所有 `Tool` 实例，负责 Schema 分发与执行调度。
- **`ToolCall`**（`base.py` 中的冻结数据类）：LLM 发出的一次工具调用的标准化表示，包含 `id`、`name`、`arguments`。

### 1.2 注册与命名规范

工具注册采用平面字典结构，但通过命名规范实现逻辑分层：

```python
self._tools: dict[str, Tool] = {}    # key = tool.name
```

**命名格式**：`namespace.action`，例如 `ctrl.commit`、`query.find_units`、`cmd.attack_move`。

此命名规范同时服务于三个目的：JSON Schema 中的函数签名、渐进式披露的命名空间过滤、以及 adapter 层的 name sanitize 逻辑。

### 1.3 read_only 属性与并发/串行策略

每个 `Tool` 子类的 `read_only` 类属性决定了其执行策略。写工具（如 `ctrl.commit`、`cmd.*`）将其设为 `False`。

`ToolRegistry.execute_calls()` 的调度策略：

1. 如果任一 tool_call 是 ctrl.commit，强制要求 calls 长度为 1
2. 检查所有工具的 read_only 标志：全为 True → `asyncio.gather(...)` 并发执行；存在 False → 线性 for 循环顺序执行

这一设计的根本原因：**状态安全性**。只读查询（obs、query）无副作用，并发安全；写操作（cmd、build、ctrl）修改游戏状态，需保证顺序性和可预测性。

```
收到 tool_calls 列表
  │
  ├─ 存在 ctrl.commit 且 len>1 → 全部返回 COMMIT_MUST_BE_SOLE_TOOL_CALL
  │
  ├─ 所有工具 read_only? 
  │    ├─ 是 → asyncio.gather 并发执行
  │    └─ 否 → for 循环顺序执行
```

### 1.4 JSON Schema 生成

`Tool.to_schema()` 返回 OpenAI function-calling 兼容格式：

```python
{
    "type": "function",
    "function": {
        "name": self.name,          # "ctrl.commit"
        "description": ...,
        "parameters": self.parameters,  # 子类定义的 JSON Schema
    }
}
```

`ToolRegistry.schemas()` 仅返回**活跃命名空间**下的工具 schema，再经 `SC2LLMAdapter._convert_tools()` 做 name sanitize（`.` 替换为 `__`）后传给 LLM。

### 1.5 执行流程

```
AgentRunner.run()
    │
    ├─ 循环调用 self._llm.chat(messages, tools=schemas)
    │   └─ LLM 返回 tool_calls
    │
    ├─ adapter 做 name unsanitize（__ → .）
    │
    └─ spec.tools.execute_calls(tool_calls)
        │
        ├─ 对每个 call:
        │   ├─ tool.cast_params(params)       ← 类型安全转型
        │   ├─ tool.validate_params(params)    ← JSON Schema 校验
        │   └─ tool.execute(**params)          ← 实际执行
        │
        └─ 返回 List[Result]
```

### 1.6 ctrl.commit 必须是唯一 tool_call

`execute_calls()` 的开头强制执行：

```python
if any(call.name == "ctrl.commit" for call in calls) and len(calls) != 1:
    return [Result.failure("COMMIT_MUST_BE_SOLE_TOOL_CALL", ...) for call in calls]
```

**原因**：`ctrl.commit` 触发游戏从 PAUSED 状态恢复运行，这是不可逆的状态转换。commit 之后当前轮次的所有其他操作都将失去意义。

## 第2章：渐进式披露机制

### 2.1 设计 vs 实现

设计文档提出了三层披露策略。实际实现修正了第二层：

| 层 | 设计文档描述 | 实际实现 |
|---|---|---|
| **第一层** | System prompt 中列出所有命名空间及其工具数量 | `_make_tool_summary()` 生成 `ns.* -- N tools` 摘要 + 引导语 |
| **第二层** | "所有工具的完整 JSON Schema 传给模型" | **已修正**——改为只传**活跃命名空间**的工具 schema |
| **第三层** | `ctrl.discover_tools("cmd")` 按需获取 | 完全一致 |

### 2.2 ToolRegistry._active_namespaces

```python
_DEFAULT_ACTIVE: set[str] = {"ctrl"}
```

初始只有 `ctrl` 命名空间活跃（4 个工具）。Agent 启动时只能看到 `ctrl.commit`、`ctrl.abort`、`ctrl.discover_tools`、`skill.load`。

### 2.3 schemas() 过滤

```python
def schemas(self) -> list[dict[str, Any]]:
    return [tool.to_schema()
            for name, tool in self._tools.items()
            if name.split(".", 1)[0] in self._active_namespaces]
```

这是披露控制的关键闸门：LLM 只能看到已激活的工具。

### 2.4 ctrl.discover_tools(namespace) 激活机制

`DiscoverToolsTool` 调用时执行：

```python
namespace = kwargs["namespace"]
added = self._registry.activate_namespace(namespace)
# 返回该命名空间下的全部工具名及其完整 JSON Schema
```

激活后该命名空间的工具在后续轮次自动可见。

### 2.5 System Prompt 中的命名空间摘要

```
obs.* -- 12 tools
query.* -- 13 tools
...

ctrl.* is always active. Before using any other namespace,
call ctrl.discover_tools("namespace") to activate it and
get the full parameter schemas. Do not guess parameter names.
```

只列命名空间和工具数量，不列具体工具名和参数，迫使 Agent 通过 `ctrl.discover_tools` 来发现细节。

## 第3章：工具通信的数据流

### 3.1 完整数据流（4 个步骤）

**Step 1 — Schema 传递**：
- `ToolRegistry.schemas()` 只返回活跃命名空间的 tool schema
- `SC2LLMAdapter._convert_tools()` 将 tool name 中的 `.` 替换为 `__`（OpenAI API 不允许函数名含点号）
- 双下划线 `__` 不会与 LLM 自然语言生成的单下划线混淆

**Step 2 — LLM 响应解析**：
- `_unsanitize_name()` 将 `__` 还原为 `.`。如果 LLM 使用了单下划线（如 `query_find_units`），通过已知命名空间前缀列表做 fallback 恢复
- arguments 为 JSON 字符串，`json.loads()` 解析。失败时包装为 `_parse_error` 结果

**Step 3 — 参数校验与执行**：

```python
args = tool.cast_params(params)       # 类型安全转型（"123"→123 等）
errors = tool.validate_params(args)   # JSON Schema 校验
data = await tool.execute(**args)     # 实际执行
return Result.success(data)           # 包装为 Result
```

**Step 4 — 结果封装与回传**：
- `AgentRunner._serialize_result()` 将 `Result` 转成 `{ok, data, error, code, meta}` 字典
- 作为 `role:"tool"` 消息追加到对话历史，供 LLM 下一轮消费

### 3.2 多 tool_calls 的并行/串行策略

| 场景 | 策略 | 说明 |
|---|---|---|
| 全 read_only | `asyncio.gather` 并发 | obs、query、hist 等可同时执行 |
| 含 write | for 循环顺序执行 | cmd、build 等需保持顺序 |
| 含 ctrl.commit | 除非是唯一 call，否则全部拒绝 | 强制最终性 |

注意：即使在串行路径中，前一个 tool 的失败**不会**中断后续工具的执行。每个工具独立执行并独立返回 `Result`。

## 第4章：Session 实测中的工具相关问题

### 4.1 工具调用频率分布

Session `20-24-24`（80 次迭代，最终 auto_abort）中呈现出极度不均匀的分布：

| 工具 | 调用次数 | 占比 | 说明 |
|------|:--------:|:----:|------|
| `ctrl.discover_tools` | 10 | 15% | 第一轮并行激活 10 个命名空间 |
| `obs.*`（观测类） | 10 | 15% | 全部调用成功，参数无错 |
| `hist.events` | 1 | 2% | 仅一次 |
| `plan.build_order` | 3 | 5% | 列模板→猜错参数→正确 |
| `plan.build_time` | 7 | 11% | 含 2 次失败的 OrbitalCommand |
| `plan.initial_state` | 1 | 2% | 一次 |
| **`plan.simulate`** | **12** | **18%** | **最高频——反复调参** |
| `query.placements` | 4 | 6% | 多次查不同建筑位置 |
| `query.find_units` | 3 | 5% | VespeneGeyser / Geyser / neutral |
| `query.closest` | 1 | 2% | 尝试找中立单位 |

对比成功 Session `18-08-06`（34 次调用，committed）：simulate 仅 3 次，timer.command 4 次，review.plan 1 次，ctrl.commit 1 次。说明工具接口的设计摩擦直接表现为 simulate 的"死循环"。

### 4.2 参数名猜错的模式

**模式 A：语义相似但参数名不同**

| 正确的参数名 | Agent 猜错的参数名 | 出现次数 | 工具 |
|-------------|-------------------|:--------:|------|
| `unit_or_building_type` | `unit_type` | 3 | `plan.build_time` |
| `name` | `opening` | 2 | `plan.build_order` |
| `type_name` | `unit_type` | 1 | `query.find_units` |

**模式 B：大小写不敏感查询**

`query.find_units(type_name="VespeneGeyser")` 返回空——底层 snapshot 中是 `VESPENEGEYSER`（大写 SC2 原始格式），`canonical_name` 不在此名称上做大写转换。

### 4.3 工具间的格式不一致（R-059）

`plan.simulate` 和 `timer.command` 的命令格式差异：

| 属性 | simulate 中的命令 | timer.command |
|------|-----------------|---------------|
| 工具名字段 | `"tool"` | `"tool_name"` |
| 参数容器 | `"args"` | `"arguments"` |

Agent 第一次 simulate 用 `"tool"/"args"` → TOOL_EXECUTION_ERROR → 此后所有 simulate 调用都手动加 `"item_name"` 顶层字段，与 timer 格式完全不同。

### 4.4 Agent 应对失败的行为模式

1. **盲试重试**（成功率 ~50%）：OrbitalCommand→"Orbital"→放弃；VespeneGeyser→"Geyser"→全 neutral→成功
2. **格式修补**（成功率 100%）：simulate 错误→加 `item_name`；INSUFFICIENT_RESOURCES→微调 at_time
3. **迂回查询**（成功率 100%）：`query.find_units` 失败→`obs.units(neutral)` 全量获取
4. **放弃**（~60% 的 session）：重试 2-3 轮后 LLM 返回空响应 → auto_abort

## 第5章：工具臃肿导致的系统性问题

### 5.1 `ctrl.discover_tools` 返回全量 schema

**现象**：Agent 并行激活 10 个命名空间，每个返回全量 JSON Schema。10 个响应合计约 20,000+ token。

**根因**：接口粒度是"命名空间"而非"具体工具"。

**影响**：20,000+ token 在 80 次迭代中持续占用上下文。按 LLM 成本计算，仅 discover 阶段的 token 约消耗 $0.06。

**解决方向**：(1) 增加 `tool_names` 可选参数；(2) 两级粒度——先摘要后详情；(3) 按调用历史动态加载。

### 5.2 `plan.initial_state` 返回冗余数据

**现象**：`producer_available_at` 字典含全部 12 个 SCV 的 24 个条目。SCV 不是生产者。

**影响**：12 次 simulate 调用，约 3000 token 冗余。

**解决方向**：过滤非生产者；或增加 `include_non_producers` 参数。

### 5.3 `obs.units` 返回全量单位细节

**现象**：12 个 SCV × 10 字段 ≈ 120 键值对。Agent 只需要计数。

**影响**：约 1500 token 浪费。

**解决方向**：`summary: bool` 参数；`fields: list[str]` 参数。

### 5.4 Agent 激活了不必要的前期命名空间

**现象**：激活全 10 个命名空间——squad(8)、cmd(18)、econ(5)、review(3) 开局无用。

**影响**：约 12,000 token 的无用 context。

**解决方向**：⓪ 步引导"开局只激活 obs/plan/build/timer"。

### 5.5 `plan.build_order` 被调 3 次才拿到模板

**根因**：双步流程 + 参数名 `name` 过于泛化。

**影响**：约 8000 token 浪费。

**解决方向**：无参时直接返回默认模板完整内容。

### 5.6 `plan.build_time` 返回 `producer_type` 不一致

**现象**：Barracks→`producer_type: "SCV"`，SCV→`producer_type: "CommandCenter"`。两者语义相同但 Agent 需映射到不同的 build 命令参数名。

**解决方向**：返回值增加 `command_pattern` 字段。

### 5.7 `query.find_units` type_name 过滤区分大小写

**现象**：`"VespeneGeyser"` ≠ `"VESPENEGEYSER"`。浪费 3 次调用。

**解决方向**：比较时统一做 `.upper()` 标准化。

### 5.8 `OrbitalCommand` 不在 costs.py 中

**现象**：`plan.build_time("OrbitalCommand")` → UNKNOWN_ITEM。人族关键升级缺失。

**解决方向**：补全 costs.py 中的升级/变形类型条目。

## 第6章：迭代分布与决策效率

### 6.1 实际 vs 理想的迭代分布

| 阶段 | 理想迭代数 | 实际（20-24-24） | 差距 |
|------|:----------:|:----------------:|------|
| 发现工具 | 1 | 1 | — |
| 获取观测 | 1-2 | 2 | — |
| 获取建造信息 | 1 | 2 | 无重试 |
| 纠错参数名 | **0** | **3** | 应为 0 |
| initial_state | 1 | 1 | — |
| placements | 1 | 2 | — |
| **plan.simulate** | **2-3** | **19** | **核心差异——6-9 倍** |
| 搜索资源 | 0 | 4 | 大小写修复后为 0 |
| timer.command | 3-4 | 0 | Agent 应在此注册 |
| review.plan | 1 | 0 | 审查步骤 |
| ctrl.commit | 1 | 0 | 提交 |
| **总计** | **12-15** | **~35** | **效率差 2.5-3 倍** |

### 6.2 耗时分析

对比两个 session：

| 指标 | Session 20-24-24（abort） | Session 18-08-06（committed） |
|------|:------------------------:|:----------------------------:|
| 工具执行时间 | ~55ms | ~35ms |
| LLM 思考时间 | ~77s | ~75s |
| 总迭代数 | 80（max_iterations） | 34 |
| 有效迭代 | ~35（44%） | ~25（74%） |
| 最终结果 | auto_abort | committed ✅ |
| **关键差异** | simulate 消耗 19 次 | simulate 仅 3 次 |

### 6.3 为什么 Agent 在 80 次迭代后仍无法 commit

三个核心原因：

1. **模拟阶段的"死循环"**：12 次 simulate 中 10 次是资源时间微调。格式完全兼容后，2-3 次足够。
2. **工具摩擦消耗迭代预算**：纠错、大小写、缺失条目等累计浪费 8-10 次迭代。
3. **LLM 空响应的"软墙"**：约 45 次迭代（56%）是空转——LLM 返回空 tool_calls，不产生行动。这是 DeepSeek v4-flash 在长上下文中的固有行为。

**结论**：修复工具接口摩擦可将有效迭代率从 44% 提升到 70-80%，但剩余的 20-30% 空转是模型本身的行为特性。工具系统的目标是让有限的有效迭代尽可能产生工具调用，而非浪费在纠错上。

## 第7章：优化方向

> 本章只列方向，不展开具体方案。详细方案见 `5.30日审查发现的问题与整改计划v2.md` 的 5.6 节及 R-056~R-059。

基于前 6 章暴露的问题，工具系统有以下优化方向：

| # | 方向 | 针对的问题 | 复杂度 |
|---|------|-----------|:------:|
| 1 | `ctrl.discover_tools` 按需返回（增加 `tool_names` 参数或 `summary_only` 模式） | 5.1：全量 schema 注入 20,000+ token | 低 |
| 2 | 过滤 `plan.initial_state` 中的非生产者条目 | 5.2：SCV 等 24 个冗余条目 | 低 |
| 3 | `obs.units` 支持摘要模式（`summary: bool`） | 5.3：全量单位细节无必要传输 | 低 |
| 4 | prompt 引导开局只激活 4 个核心命名空间 | 5.4：squad/cmd/econ 开局无用 | 低 |
| 5 | `plan.build_order` 无参返回默认模板完整内容 | 5.5：3 次调用才拿到模板 | 低 |
| 6 | 统一 type_name 大小写比较 | 5.7：`VespeneGeyser` vs `VESPENEGEYSER` | 低 |
| 7 | 补全 `costs.py` 缺失的升级/变形单位 | 5.8：OrbitalCommand UNKNOWN_ITEM | 低 |
| 8 | 工具返回值增加 `verbose` 参数以支持精简模式 | 返回体普遍过大 | 中 |

## 第8章：选定方案——SubAgent 代理 + 命名空间级权限

### 8.1 方案概述

**核心思想**：主 Agent 不再直接操作 85 个工具，而是通过少数 SubAgent 代理。主 Agent 在创建 SubAgent 时声明任务，SubAgent 内部自己决定调用哪些工具、以何种粒度返回数据。主 Agent 的工具面从 85 个缩减到约 15 个（控制出口 + 极轻量查询）。

**权限模型**：命名空间即权限。`ToolRegistry.filter(namespaces={"obs", "query", "hist"})` 一行代码裁剪出只含指定命名空间的子注册表，注入 SubAgent。不需要 RBAC、角色表、ACL——工具命名规范已经天然完成了权限划分。

### 8.2 两个 SubAgent + 主 Agent 亲自执行

| 角色 | 代理的工具 | 职责 | 核心价值 |
|------|:--------:|------|------|
| **观测 SubAgent** | obs(12) + query(13) + hist(5) = 30 | "给我当前局势概览""聚焦敌方单位" | 内部决定摘要 vs 全量，主 Agent 不再接触原始观测数据 |
| **规划 SubAgent** | plan(4) | "基于模板和初始状态生成时间线" | at_time 二分搜索替代 LLM 猜参数，内部 simulate 调参不暴露 |
| **主 Agent 亲自执行** | `timer.command` / `timer.monitor` | "我拿到规划结果后自己注册每一个命令" | **执行权不下放——主 Agent 对每一个命令签字** |

**subAgent 不是一次性外包，而是主 Agent 驾驶的迭代探索工具。** 主 Agent 的思考循环没变——它仍然是探索→接收→再思考的驱动者。变化在于：以前主 Agent 每次思考后要亲自调 3-5 个 obs/query 工具、读原始数据、自己判断；现在它把"调什么工具、以什么粒度返回"的脏活委托给 subAgent，自己专注于解读结果和决定下一步。

**观测的迭代模式**：
```
主 Agent："了解当前局势" 
  → dispatch.observe("全局概览")
  ← subAgent："12 SCV，1 CC，敌方2 Zergling在右侧，矿物400"
主 Agent："右侧有狗？上次没看到。聚焦右侧敌人的详细信息。"
  → dispatch.observe("右侧敌方单位的类型、tags、最早观测时间、移动方向")
  ← subAgent："2 Zergling，tags=xx/yy，1:30首次出现，正在接近矿区"
主 Agent："它们在接近矿区。信息够了，开始规划。"
```
每次 dispatch，subAgent 都会根据任务描述自己决定调 `obs.enemy_visible` 还是 `query.find_enemy` 还是 `hist.unit`，并自己决定返回摘要还是全量。主 Agent 始终是思考者，只是不再亲手翻原始数据。

**规划的迭代模式**：
```
主 Agent："基于 terran_1rax_expand 生成 90 秒时间线"
  → dispatch.plan(...)
  ← subAgent："时间线生成完毕，Barracks 于 t=78 完成"
主 Agent："Barracks 太晚了，提前到 60 秒前完成"
  → dispatch.plan("调整：Barracks 在 t=60 前完成，其他不变")
  ← subAgent："调整后：将 SCV#14 提前到 t=22，Barracks 在 t=58 完成"
主 Agent："可以，采纳这个时间线。"
```
subAgent 负责调 `plan.build_time`→`plan.simulate`→调整→再模拟的循环，主 Agent 只需看结果、说"再早点"或"可以了"。

**执行不下放的原因**：观测和规划是信息收集和纯计算——可以安全代理。执行是改变游戏状态——建什么、何时建、用哪个 SCV、在哪个坐标——必须由主 Agent 最终决定。规划 subAgent 输出的命令列表供主 Agent 审阅，主 Agent 认可后亲手逐条 `timer.command` 注册。

### 8.3 主 Agent 保留

| 保留原因 | 工具 |
|----------|------|
| 决策出口，必须主 Agent 签字 | `ctrl.commit`, `ctrl.abort` |
| 亲自执行 | `timer.command`, `timer.monitor` |
| 极轻量查询，不值得 spawn | `obs.game_time`, `obs.resources`, `obs.enemy_visible`, `plan.build_time`, `query.can_afford`, `timer.list`, `hist.events` |
| 调度 subAgent | `dispatch.observe`, `dispatch.plan` |
| 审查 | `review.plan` |
| 技能 | `skill.load` |

主 Agent 工具面：**~16 个**（从 85）。

### 8.4 最小实施路径

**第一步：`ToolRegistry.filter(namespaces)` — 权限的核心**

```python
# sc2_agent/tools/registry.py 新增方法，~10 行
def filter(self, *, namespaces: set[str]) -> ToolRegistry:
    sub = ToolRegistry()
    for name, tool in self._tools.items():
        if name.split(".")[0] in namespaces:
            sub.register(tool)
    return sub
```

**第二步：`dispatch.observe` 和 `dispatch.plan` — 两个调度工具**

各自约 50 行。内部创建 SubAgent → `run(task)` → 返回结构化 JSON。观测用 `filter(namespaces={"obs","query","hist"})`，规划用 `filter(namespaces={"plan"})`。

**第三步：修改主 Agent 系统提示词**

- ② 粗看 → 改用 `dispatch.observe("当前局势概览")`，不再引导去 `ctrl.discover_tools("obs")`
- ④ 时间线推理 → 改用 `dispatch.plan("基于 terran_1rax_expand 生成时间线")`，不再引导 Agent 自己反复调 `plan.simulate`
- 主 Agent 拿到规划结果后，亲自用 `timer.command` 逐条注册

### 8.5 预期效果

| 指标 | 当前 | 选定方案后 |
|------|:----:|:------:|
| 主 Agent 工具面 | 85 | ~16 |
| simulate 主 Agent 迭代 | 12-19 | 1（dispatch.plan） |
| 决策周期 token | ~200K | ~60-80K |
| 参数猜错 | 每轮 2-5 次 | 接近消除 |
| 执行权 | — | 不下放，主 Agent 亲手注册每一个 timer.command |

### 8.6 原则

- **信息收集和计算可以代理，执行权不下放**
- **不做 SubAgent 发现机制**：主 Agent 创建 SubAgent 时描述任务，不是从列表中选择
- **不做细粒度权限**：命名空间级够用，`filter(namespaces={...})` 一行搞定
- **不做 SubAgent 之间的直接通信**：主 Agent 作为中枢转手，保持决策链可审计
- 架构改动量：**~150 行新增，0 行改动现有代码**
