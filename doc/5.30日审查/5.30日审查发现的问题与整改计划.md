### R-026 | P0 | `econ.set_mining` 工具缺失

**来源**：第2轮审查 `sc2_agent/tools/econ.py`

**设计文档要求**（FR-05）：
```
econ.set_mining — (base_id, gas_count?) 调整某基地气矿工人数
```

**当前状态**：未实现。econ 命名空间只有 4 个工具（`gather`、`transfer_workers`、`expand`、`build_gas`），缺少第 5 个。

**影响**：Agent 无法精细控制气矿工人分配，影响 gas 收入优化。

**整改方向**：
- 新建 `EconSetMiningTool`，参数 `base_id: int` + `gas_count: int | None`
- 通过 burnysc2 BotAI 的 workers 集合分配气矿工人
- 无气矿工人数限制时默认每气矿 3 人

### R-027 | P0 | `squad.list` 工具缺失

**来源**：第2轮审查 `sc2_agent/tools/squad.py`

**设计文档要求**（FR-05）：
```
squad.list — 当前所有小队的摘要
```

**当前状态**：squad 命名空间有 5 个工具（`create`、`add`、`remove`、`disband`、`order`），缺少 `list`。

**影响**：Agent 创建多个小队后无法列出它们，只能凭记忆管理，容易遗漏。

**整改方向**：
- 新建 `SquadListTool`，返回所有小队名 + 成员数 + 成员 tag 列表
- 依赖当前 squad 字典注入

### R-029 | P1 | `obs.resources` 缺少收入字段

**来源**：第2轮审查 `sc2_agent/tools/obs.py`

**设计文档示例**：
```json
{"minerals":820, "gas":45, "supply":"32/46", "income_min":450, "income_gas":80}
```

**当前实现返回**：
```json
{"game_time": ..., "minerals": ..., "gas": ..., "supply_used": ..., "supply_cap": ..., "supply_available": ...}
```
缺少 `income_min` 和 `income_gas`。

**影响**：Agent 做经济规划时只能用静态成本表（COSTS），无法根据当前实际收入速率做精确计算。`plan.simulate` 在 `bot._make_simulation_state()` 中自行估算了 `mineral_income_rate` 和 `gas_income_rate`，但 Agent 通过 `obs.resources` 看不到这些值。

**整改方向**：
- 在 `ObsResourcesTool.execute()` 中增加 `income_min` 和 `income_gas` 字段
- 从 `snapshot` 或 BotAI 获取（bot.py 的 `_make_simulation_state()` 已有计算逻辑，可复用）

### R-030 | P1 | `review.params` 不验证 tag 真实存在

**来源**：第2轮审查 `sc2_agent/tools/review.py`

**当前行为**：`ReviewParamsTool._has_producer_tag()` 只检查 `arguments` 字典中是否有 `producer_id` / `structure_tag` / `worker_tag` 等 key，但**不调用 `bot.find_by_tag()` 验证该 tag 是否真的存在于当前游戏状态**。

**设计文档要求**（FR-08）：
> review.params 必须纯代码检查 tag、坐标、放置位置、生产建筑、科技前置、资源和冲突。

**影响**：
- 假阳性：tag 格式正确但实际不存在 → `review.params` 返回 PASS
- `review.plan` 内部调用了 `plan.simulate` 做模拟，资源层面的冲突会被捕获
- 但 tag 层面的假阳性只能靠 timer 执行时发现（已经晚了）

**整改方向**：
- 给 `ReviewParamsTool` 注入 BotAI（或 ObservationProvider）
- 对每个 tag 参数调用 `find_by_tag()` 验证存在性
- 或在 `review.plan` 层面统一做 tag 校验

### R-031 | P2 | Session 写入方式

**来源**：第1轮审查 `sc2_agent/agent/session.py`

**当前行为**：`SessionManager.save()` 使用 `mode="w"` 完整覆写整个 JSONL 文件。

**设计文档要求**（FR-10）：
> 原始 messages 不得删除或重写（追加写入，永不删除）

**影响**：如果进程在 save() 中途崩溃，整个 session 文件可能损坏。正常运行时不影响功能（所有消息都在内存中然后全量写出）。

**整改方向**：
- 改为 `mode="a"` 追加单条消息
- 或保留当前全量覆写但加入 `temp + rename` 原子写入（类似 `MemoryStore.save_json()` 的做法）

### R-032 | P2 | 整合 LLM 使用主模型

**来源**：第3轮审查 `sc2_agent/bot.py` + `sc2_agent/memory/consolidator.py`

**当前行为**：`bot.py:282` 中 MemoryConsolidator 的 provider 是 `_LLMConsolidationProvider(self._llm_adapter)`，使用的就是主 Agent 的 LLM。

**设计文档要求**（6.3节）：
> MemoryConsolidator 使用便宜 LLM，目标延迟 1-2 秒

**影响**：每次 token 超预算时调用主模型做整合，token 消耗和延迟都比用便宜模型高。

**整改方向**：
- 在 `SC2LLMAdapter` 基础上增加第二个 sticky client（如 qwen-turbo）
- 或新建 `CheapLLMAdapter` 专门给 Consolidator 用

### R-033 | P2 | Token 估算粗糙

**来源**：第3轮审查 `sc2_agent/memory/consolidator.py`

**当前行为**：`MemoryConsolidator._estimate_tokens()` 使用 `len(text) // 4`。

**技术栈建议**（文档 9.1）：
> tiktoken>=0.12 — Token 计数

**影响**：英文场景下 `//4` 近似可用，但中英混合（prompt 中有中文）偏差可达 50%+。整合 LLM 的触发时机不准确——可能过早触发（浪费调用）或过晚触发（上下文超限）。

**整改方向**：
- 引入 `tiktoken`，使用对应模型的编码器做精确计数
- 或至少对中文使用 `len(text) // 2`，对中英文分段估算

### R-035 | P2 | `econ.transfer` 语义偏差

**来源**：第2轮审查 `sc2_agent/tools/econ.py`

**设计文档定义**（3.3节）：
```
econ.transfer — (count, from_base_id, to_base_id) 跨基地转移工人
```

**当前实现**：`EconTransferWorkersTool(worker_tags, resource_tag)` — 将指定 tag 的工人转移到指定资源。

**差异**：文档是按数量的宏观操作（"从基地 A 转移 3 个工人到基地 B"），实现是按 tag 的微观操作（"让 tag=10,11,12 去采集 tag=200 的资源"）。

**影响**：语义偏差不影响功能正确性（Agent 可以先 `query.find_workers` 拿到 tag 再调 `transfer_workers`），但操作更繁琐。当前实现更接近 `econ.gather`。

**整改方向**：
- 将现有工具重命名为 `econ.gather`（已完成）
- 新建 `EconTransferTool(count, from_base_id, to_base_id)` 实现文档语义
- 或保持现状，在工具描述中说明用法

### R-036 | P0 | timer 命令执行失败不唤醒 Agent ⭐新发现

**来源**：2026-05-30 执行模型分析

**场景**：
```
t=120s: Agent 决策，给 SCV(tag=42) 下达 build.structure(SupplyDepot, 35, 40)
        → timer.command(at_time=127.0, ...)
t=125s: SCV(tag=42) 被敌方死神击杀
t=127s: TimerScheduler 执行命令
        → _resolve_units(bot, [42]) → TAG_NOT_FOUND
        → 命令标记为 "failed"，写入 run_history
        → Agent 继续休眠 ← 问题在此！
```

**当前行为**：

| 层级 | 处理方式 | 是否唤醒 Agent |
|------|---------|:---:|
| `cmd.py` `_resolve_units()` | 返回 `TAG_NOT_FOUND` + `ok=False` | — |
| `scheduler.py` `tick()` | 标记 status="failed"，写 run_history | — |
| `bot.py` `_tick_scheduler()` | `_logger.debug()` 打日志 | ❌ 不唤醒 |
| `SchedulerResult` | 只有 `executed` 列表，不区分成功/失败 | ❌ 无失败信息 |

**影响**：
1. **Agent 完全不知情**：Agent 要等到下一次 monitor 触发才能通过 `timer.list` 或 `hist.events` 发现失败
2. **级联失败**：假设 SCV(tag=42) 死亡 → `build.structure(SupplyDepot)` 失败 → 30 秒后 `build.train(Marine)` 需要补给空间 → 也失败 → Agent 对全部 10 个级联失败一概不知
3. **静默降级**：游戏正常进行，但 Agent 的决策无法落地

**根因**：
- `SchedulerResult` 不区分 executed 中的成功和失败
- `_tick_scheduler()` 没有检查失败并调用 `wake_to_thinking()`
- 设计文档 FR-06 没有规定"命令执行失败时是否唤醒 Agent"

**整改方向**：

```
1. SchedulerResult 增加 failed: list[str] 字段
2. tick() 中收集失败命令ID
3. bot._tick_scheduler() 检测到 failed 非空时 → wake_to_thinking()
4. 唤醒时附带 reason: "N commands failed, including: cmd.move(tag=42 TAG_NOT_FOUND)"
```

### R-037 | P0 | SquadCreateTool 与其他 squad 工具使用不同 dict ⭐阻断性Bug

**来源**：5.30 并行审查 — `squad.py` + `bot.py:429-434`

**根因**：`bot.py` 中局部创建 `squads = {}` 传入 `SquadAddTool/RemoveTool/DisbandTool/OrderTool`，但 `SquadCreateTool.__init__` 内部自建 `self._squads = {}`，两者完全隔离。

**影响**：`squad.create(name="army1")` 写入 SquadCreateTool 内部字典；`squad.add(squad_name="army1")` 在空的局部字典中查找 → SQUAD_NOT_FOUND。**所有 squad 操作不可用。**

**整改方向**：共享同一个 dict，传入所有 squad 工具包括 SquadCreateTool。

### R-038 | P1 | cmd.py 所有 18 个工具动作执行阶段无 try/except

**来源**：5.30 并行审查 — `cmd.py` 全部工具

**根因**：`_resolve_units()` 检查 tag 存在后，`for unit in units: unit.move(...)` 等动作调用没有 try/except。若单位在 _resolve_units 返回后、动作执行前死亡，异常穿透。

**正确模式参考**：`BuildStructureTool` (build.py:125-126) — `except Exception as e: return _result(False, [{"code": "BUILD_ERROR", "detail": str(e)}])`

**整改方向**：18 个 cmd 工具的动作调用包裹 `try/except Exception as e`，失败返回结构化错误码。

### R-039 | P1 | BuildTrainTool 部分成功时返回 ok=True 且不告知期望 count

**来源**：5.30 并行审查 — `build.py:191-203`

**根因**：训练循环 `for _ in range(count)` 中 `except Exception: break` 吞掉异常，返回 `ok=True, trained=2`。Agent 不知道期望训 5 个。

**整改方向**：返回值增加 `requested: count` 字段；异常时记录到 errors。

### R-042 | P1 | review.logic SubAgent 失败伪装成 success+WARN

### R-050 | P1 | Agent 混淆 timer.monitor 和 timer.command 的职责 ⭐实测发现

**来源**：2026-05-30 系统实测

**现象**：Agent 输出了以下推理和工具调用：
> "模拟显示收入模型不够精确。我用定时监控的方式来处理，当矿物达到 100 时自动建造补给站。"

然后调用：
```json
timer.monitor({"metric":"minerals","op":">=","value":100,"reason":"有足够矿物建造 SupplyDepot"})
```

**根因**：Agent 将 `timer.monitor` 误解为"条件触发执行器"——以为设置了 monitor 后，条件满足时系统会自动建造补给站。实际上 `timer.monitor` 的唯一作用是**暂停游戏、唤醒 Agent**。建造动作必须通过 `timer.command(at_time=..., tool_name="build.structure", arguments={...})` 来排期。

**为什么系统没有阻止这个错误**：
- `timer.monitor` 的参数完全合法——metric/minerals、op/>=、value/100 都是有效的
- 系统不会校验"你是否还差一个 timer.command 来完成实际建造"
- 当前 system prompt 和 main-flow skill 都没有清晰区分这两个工具的职责边界

**影响**：Agent 以为补给站已经安排好了，实际只是设了一个"矿到 100 就叫醒我"的条件。补给站永远不会被建造，Agent 要到下次醒来通过 `timer.list` 检查才能发现——如果它记得检查的话。

**整改方向**：
1. 在 system prompt 第 1 段和 main-flow skill 中明确区分：
   - `timer.command` = "在 at_time 时刻执行指定工具"（命令执行器）
   - `timer.monitor` = "当条件满足时暂停游戏并唤醒 Agent"（条件唤醒器）
2. 在 main-flow skill 反面示例中增加"用 monitor 替代 command"的错误模式
3. 在 `_make_skill_summary()` 或 tool description 中强调两者的职责差异

### R-051 | P0 | WORKER_TYPE_NAMES 大小写不匹配 ⭐实测发现

**来源**：2026-05-30 系统实测 Session 分析

**根因**：`query.py:297` 定义 `WORKER_TYPE_NAMES = frozenset({"scv", "probe", "drone"})`（全小写），但 `canonical_name("SCV")` 返回 `"SCV"`（大写，来自 ALIASES 映射）。`"SCV" in {"scv", "probe", "drone"}` → `False`。12 个 SCV 全部被滤掉。

**影响**：`query.find_workers` 在任何情况下都返回空列表。Agent 无法查询工人状态、无法获取空闲工人 tag、无法筛选采矿/采气工人——这些都是建造操作的前置步骤。

**整改方向**：`WORKER_TYPE_NAMES` 改为 `frozenset({"SCV", "Probe", "Drone"})`，与 `canonical_name()` 输出一致。

### R-052 | P0 | BurnySC2ObservationAdapter 不读取中立单位 ⭐实测发现

**来源**：2026-05-30 系统实测 Session 分析

**根因**：`burnysc2_adapter.py:snapshot()` 只读取 `bot.units`（我方单位）和 `bot.structures`，不读取 `bot.mineral_field`、`bot.vespene_geyser`。导致：
- `query.find_units(MineralField)` 永远返回空
- `query.find_units(VespeneGeyser)` 永远返回空
- `obs.unit(resource_tag)` 永远返回 TAG_NOT_FOUND

但 `bot.find_by_tag` 本身是正常的（搜索范围包含 mineral_field/vespene_geyser），只是 `obs.unit` 工具通过快照而非 bot 查找。

**影响**：Agent 无法查询矿脉和气矿的位置和数量，无法在 simulate 中获取资源分布信息，也无法通过 `query.closest` 找到最近的矿脉。

**整改方向**：`BurnySC2ObservationAdapter.snapshot()` 中追加读取 `bot.mineral_field`、`bot.vespene_geyser`，以 `alliance="neutral"` 加入 `ObservationSnapshot.units`。同时 `ObsUnitTool` 可增加 bot 回退查找。

### R-053 | P1 | plan.simulate 默认 income_rate=0 ⭐实测发现

**来源**：2026-05-30 系统实测 Session 分析

**根因**：`simulator.py:SimulationState.from_dict()` 使用 `data.get("mineral_income_rate", 0.0)` 默认值。Agent 调用 `plan.simulate` 时只传 `{minerals, gas, supply_used, supply_cap}`，缺 income_rate → 默认 0 → 模拟收入恒为 0 → 全部门报 INSUFFICIENT_RESOURCES。

深层根因：Agent 没有获取 income_rate 的途径——`obs.resources` 不返回 income（R-029），`plan.simulate` 不自动补全。`bot._make_simulation_state()` 内部能正确计算收入率（SCV 数×0.75），但该逻辑只暴露给 `review.plan` 而不暴露给 Agent。

**影响**：Agent 无法正确使用 `plan.simulate`——要么收入为 0 全部失败，要么放弃 simulate。本 session 中 Agent 看到失败后直接改用 monitor 兜底，放弃了规划。

**整改方向**：推荐新增 `plan.initial_state` 工具——内部调用 `bot._make_simulation_state()` 逻辑，返回完整的 SimulationState（含 income_rate、units、structures、producer_available_at）。Agent 先调 `plan.initial_state()` 再调 `plan.simulate()`。同时在 simulate 的 description 中说明需要 income_rate 字段。

### R-054 | P1 | PAUSED_THINKING 中允许直接执行写操作 ⭐实测发现

**来源**：2026-05-30 系统实测 Session 分析

**根因**：Agent 在 PAUSED_THINKING 期间直接调 `build.train(SCV)` 并立即执行成功，没有通过 `timer.command` 排期。设计文档规定所有写操作必须通过 timer.command → Timer Scheduler 在 RUNNING_SLEEP 执行。但当前 ToolRegistry 不阻止任何工具在 PAUSED_THINKING 中被直接调用。

虽然 t=0 时立即执行 build.train 和排期 `timer.command(at_time=0, ...)` 的功能效果相同（Timer Scheduler 第一帧就执行），但流程纪律上的差异显著：即时执行绕过 staging、绕过审查、绕过 simulate——Agent 后续的 simulate 必须手动把"已执行"的命令补进 command list，容易出错。

**影响**：Agent 可能养成"能立即做的就立即做"的习惯，staging 和审查机制逐渐被架空。

**整改方向**：在 system prompt 第 1 段和 main-flow skill 中明确规定：**PAUSED_THINKING 中只允许 obs.* / query.* / plan.* / hist.*（只读）和 timer.command / timer.monitor（排期）**。任何写操作（build.* / cmd.* / econ.* / squad.*）必须通过 timer.command 排期。同时可在 ToolRegistry 层增加状态校验——PAUSED_THINKING 中拒绝直接执行 write 工具。

### R-055 | P1 | review.plan 不检查模板目标完成度 ⭐实测发现

**来源**：2026-05-30 系统实测 Session 分析

**根因**：staging 中只有 1 个 monitor(minerals>=100) + 1 个 timer.command(build.train)，完全没有 SupplyDepot 建造命令。`terran_1rax_expand` 模板 step 2 明确要求造 SupplyDepot。但 `review.plan` 返回 `verdict: PASS, issues: []`——它只检查了资源是否够用、坐标是否合法，没有检查"模板的目标是否被覆盖"。

审查器的四个维度（对抗合理性、资源合理性、战略一致性、时间线合理性）中，"战略一致性"理论上应该暴露这个问题——模板说要 SupplyDepot，staging 里没有——但当前 subAgent 可能没有收到模板内容作为审查上下文。

**影响**：审查形同虚设——最关键的遗漏（核心建造命令缺失）没有被标记。Agent 相信审查结果，以为一切就绪，实际什么都没安排。

**整改方向**：`review.plan` 在构建审查任务时，将当前选定的 build_order 模板内容传入审查 subAgent，增加"模板覆盖度"维度的检查。
