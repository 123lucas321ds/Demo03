## 1. 开发总原则

- 不启动游戏作为早期开发依赖。
- 不创建新 conda 环境，使用已有 `LLM` 环境。
- 涉及 burnysc2 API 时，只阅读本地库。
- 每个阶段必须能独立运行对应单元测试。
- 先做纯代码可测模块，再接 BotAI。
- 任何阶段没有测试通过，不进入下一阶段。
- 最小闭环优先于工具数量，不先实现全部 83 个工具。

## 2. 阶段计划（Phase 0–12）

### Phase 0：项目骨架与基础设施

目标：建立目录结构和基础配置，不接入真实 SC2。

交付：`runtime/state.py`、`config/settings.py`、`logging/logger.py`、基础异常/结果模型、`tests/` 结构。

测试：配置默认值、runtime state 合法/非法转换、日志模块初始化、基础 import。

### Phase 1：Tool 基类与 ToolRegistry

交付：`tools/base.py`、`tools/registry.py`、schema cast/validate、读写工具分类（读并发、写串行）、`ok/data/error/code/meta` 统一结果。

测试：参数校验、类型转换、enum/range/array/object 校验、未注册工具错误、读写顺序、`ctrl.commit` 独占。

### Phase 2：AgentRunner 与 Session

交付：`agent/runner.py`、`agent/session.py`、assistant tool_calls / tool result 结构化追加、`last_consolidated` 游标、`_find_legal_start()` orphan 裁剪、max_iterations 自动 abort。

测试：fake LLM tool loop、tool_call_id 绑定、history 裁剪、游标持久化、max_iterations abort。

### Phase 3：MemoryStore 与 game_state

交付：`memory/models.py`、`memory/store.py`、`memory/renderer.py`、`game_state.json` 权威源 + `game_state.md` 派生视图、原子保存 JSON。

测试：JSON 初始化/更新、Markdown 渲染、JSON 更新后 Markdown 同步、Markdown 不作为写入源。

### Phase 4：Timer Staging、staging_hash 与 Commit 顺序

交付：`timer/models.py`、`timer/staging.py`、`timer/store.py`、`ctrl.commit`/`ctrl.abort`、SHA256 staging hash、commit 顺序（snapshot/events → session → game_state → consolidator → markdown → timer → sleep）。

测试：hash 稳定、hash 变化、未 review 拒绝 commit、hash 不匹配拒绝、commit 顺序、失败时 timer 不注册、abort 清空 staging。

### Phase 5：MemoryConsolidator 同步整合

交付：`memory/consolidator.py`、token 预算判断、wake-up 周期边界 chunk 选择、整合 LLM 调用、游标推进、失败降级。

测试：未超预算不整合、超预算触发、只修改战略判断/当前优先级、成功推进游标、失败不推进游标、整合后重新渲染 Markdown。

### Phase 6：plan.simulate 确定性模拟器

交付：`planning/simulator.py`、`planning/costs.py`、`planning/tech_tree.py`、`plan.simulate(commands, horizon)`、纳入 staged commands + active timers + 生产队列 + 补给 + 建筑占用 + 工人占用、返回 points/assumptions/first_failure。

测试：资源增长、建造扣矿、训练扣矿和补给、资源/补给不足、建筑占用冲突、科技缺失、active timers 合并。

### Phase 7：obs/query 工具骨架

交付：`tools/obs.py`、`tools/query.py`、`observation/collector.py`、BotAI 协议接口/fake fixture。

测试：obs.resources、obs.units、obs.structures、query.find_units、query.idle_producers、query.can_afford、query.tech_requirement。

### Phase 8：timer.command 执行与 Scheduler

交付：`timer/scheduler.py`、到期 command 执行、monitor 条件评估、before_time 过期、monitor 触发切回 PAUSED_THINKING。

测试：到期执行、未到期不执行、失败记录 run_history、monitor 触发 wake、before_time 过期、状态约束。

### Phase 9：review.params 与 review.plan

交付：`tools/review.py`、`review.params`、`review.plan(staging_hash)`、plan.simulate 联动验证、review hash 绑定。`review.logic` 可暂留 stub。

测试：tag/坐标/资源/补给/科技/冲突检查、review 后 staging 变化拒绝 commit、同 hash 允许。

### Phase 10：最小闭环集成测试

交付：fake LLM、fake BotAI、fake clock。完整 wake → tool calls → staging → review → commit → sleep → timer → monitor → wake。

测试：全流程不依赖 SC2。

### Phase 11：真实 burnysc2 Adapter 核对

不启动游戏，只读核对本地 burnysc2 源码，校准字段名，写 adapter 层。

测试：fake burnysc2 对象字段兼容。

### Phase 12：后续真实游戏接入

实际启动 SC2、BotAI `on_step` 接入、stop-the-world 验证、VeryEasy AI 对局。详见下方阶段 A/B/C/D 计划。

## 8. 前端设计：对局复盘查看器

### 8.1 架构

单文件 HTML（`sc2_agent/viewer.html`），包含内嵌 CSS + JS。零后端依赖，浏览器直接打开。

### 8.2 数据加载

- 拖拽 `app.log` 文件到页面 → FileReader 解析 JSON 行
- 每行解析为 `{time, level, event, game_time, wake_id, message, ...}` 
- 存入内存数组，按 `game_time` 排序

### 8.3 面板一：时间线总览

Canvas 自绘。横轴 = `game_time`（0 到 max）。纵向分层：

| 层 | 内容 | 渲染 |
|----|------|------|
| L1 Agent | wake 周期 | 矩形色块，绿=committed, 黄=aborted, 红=error；宽度=周期跨度 |
| L2 LLM | LLM 调用 | 竖线，粗细 ∝ latency_ms；悬停 tooltip（model, tokens, 耗时）|
| L3 Tools | 工具调用 | 圆点，颜色编码 ns（绿=obs/query, 橙=cmd/build/econ, 蓝=plan/review, 灰=ctrl/timer）|

点击任意节点 → 面板三自动过滤到对应 wake_id。

### 8.4 面板二：资源与状态曲线

Chart.js 折线图。三条线：minerals（蓝）、vespene（绿）、supply_used（红）。

数据来源：遍历日志中 `event === "game.state"` 的记录，提取 `minerals`、`gas`、`supply_used` 字段。

图表下方标注 Agent 决策点：虚线标记 commit/abort 事件时间。

### 8.5 面板三：事件详情列表

表格列：game_time、level、event、message。上方过滤栏：
- 按 wake_id 过滤
- 按 event 类型过滤（下拉多选）
- 按 level 过滤（WARNING+）
- 文本搜索

### 8.6 文件结构

```
front/
└── viewer.html   ← 唯一交付物，约 800 行
storage/
└── 2026-05-30_13-38-52/   ← 对局数据目录
```

`front/` 与 `storage/` 同级，HTML 拖拽加载 `../storage/{game_id}/logs/app.log`。

### 8.7 技术选型

| 组件 | 方案 |
|------|------|
| 折线图 | Chart.js CDN（`cdn.jsdelivr.net`） |
| 时间线 | 纯 Canvas 自绘 |
| 表格 | 原生 `<table>` + DOM 操作 |
| 样式 | 内嵌 CSS，暗色主题 |
| 外部依赖 | 仅 Chart.js 一个 CDN 引用 |
