# SC2 Agent 重构需求文档

> 来源：`doc/memory-centric-agent-redesign.md`
> 目标：将初始设计中的目标、边界、约束、功能和验收标准整理为可执行的需求规格。

## 1. 背景

当前 SC2 Agent 采用 Strategy -> Phase -> Executor -> Supervisor 的三层流水线。该设计假设 LLM 能在一次请求中消化全量观测、稳定输出结构化计划，并通过外部 Supervisor 保障质量。实际运行中出现了明显退化：

- 全量观测导致模型倾向输出安全默认动作，例如连续造补给站而忽略关键产能。
- Supervisor 造成角色混淆，执行层开始模仿审查输出，带来解析失败和职责漂移。
- 历史摘要只是数据拼接，不构成可持续演化的游戏记忆。
- 固定轮询频率既浪费 LLM 调用，也无法在执行偏差或突发战斗时及时唤醒。

本次重构要求彻底放弃旧三层流水线，采用 nanobot 风格的单 Agent 工具驱动架构：一个主 Agent、一组原子化工具、一套持续记忆、一个游戏内 Timer Scheduler。

## 2. 目标

### 2.1 业务目标

- 让 Agent 像 SC2 人族指挥官一样按需观察、推理、行动，而不是一次性生成 YAML 或文本计划。
- 让战略判断持续沉淀在 `game_state.json` 中，并派生为 `game_state.md` 注入上下文，而不是每轮从零选择策略。
- 让 Agent 自主决定下一次唤醒时间和异常唤醒条件。
- 让时间线规划由主 Agent 在工具调用循环中逐步推算完成，并通过 `timer.command` 交给调度器执行。
- 让审查从外部强制 Supervisor 改为主 Agent 可主动调用的 `review.*` 工具。

### 2.2 技术目标

- 使用标准 OpenAI function calling 消息结构，保留 assistant/tool message 的结构化关系。
- 建立约 83 个原子化工具，按命名空间渐进式披露。
- 建立 JSONL Session、事件日志、三层快照、`game_state.json` 权威源和 `game_state.md` 派生视图组成的轻量记忆系统。
- 引入 MemoryConsolidator，通过 token 预算触发旧消息整合。
- 保留可复用的游戏基础设施，如 ProductionManager、SquadManager、observation、DecisionLogger 等。

## 3. 范围

### 3.1 本期范围

- AgentRunner 工具调用循环。
- ToolRegistry 与工具基类。
- `obs.*`、`query.*`、`cmd.*`、`build.*`、`econ.*`、`squad.*`、`timer.*`、`plan.*`、`review.*`、`hist.*`、`ctrl.*` 命名空间。
- Timer Scheduler 游戏内定时命令和条件唤醒。
- Session JSONL、events JSONL、botAI 快照、`game_state.md`。
- MemoryConsolidator 与 `last_consolidated` 游标。
- System prompt 重建。
- Skill 加载机制，支持 `skill.load(name)`。
- 旧 StrategyLayer、PhaseLayer、ExecutionLayer、Supervisor 和 TriggerScheduler 的替换与清理。

### 3.2 后续范围

- `exec.*` 沙盒执行器。
- 更复杂的 monitor 校准 subAgent。
- 更细粒度的差异分析 subAgent。
- 工具热加载或 MCP 化工具系统。

### 3.3 非目标

- 不重新安装 StarCraft II。
- 不创建新的 conda 环境，继续使用已有 `LLM` 环境。
- 不保留旧三层流水线作为并行主路径。
- 不把策略池 YAML 继续作为每轮策略选择入口。
- 不在 System Prompt 中塞入大量领域知识或固定流程。
- 不实现图灵完备的 timer 表达式引擎。

## 4. 运行环境约束

- 游戏环境：StarCraft II 已在本机安装。
- Python 环境：使用 `D:\Anaconda\anaconda\envs\LLM\`。
- burnysc2 路径：`D:\Anaconda\anaconda\envs\LLM\Lib\site-packages\sc2\`。

- 本项目路径以当前工作区为准。

## 5. 用户与角色

| 角色 | 职责 |
|------|------|
| 主 Agent | 观察、推理、规划时间线、调用审查、提交命令、设定唤醒条件 |
| Timer Scheduler | 在游戏帧循环中执行定时命令并评估唤醒条件 |
| MemoryConsolidator | 维护 `game_state.md`，在 token 超预算时整合旧消息 |
| review Sub-Agent | 从独立上下文审查命令列表的战术与战略质量 |
| 差异分析 Sub-Agent | 对比当前观测与历史快照，返回结构化变化 |
| monitor 校准 Sub-Agent | 高频唤醒时调整 monitor 阈值 |
| 工具 Handler | 封装 burnysc2、管理器和存储层能力，返回结构化结果 |

## 6. 功能需求

### FR-00 Stop-the-world 执行模型

- 系统必须采用 stop-the-world 模型：LLM 思考、工具规划、Sub-Agent 审查期间游戏完全暂停。
- Agent 只有在 `ctrl.commit(staging_hash)` 成功后进入休眠，游戏才恢复运行。
- Timer Scheduler 只允许在 Agent 休眠、游戏运行期间执行到期命令和评估 monitor。
- `timer.monitor` 触发后必须立即暂停游戏，并以触发 reason 唤醒 Agent。
- `ctrl.commit` 必须是最后一轮唯一的 tool_call；commit 所在响应中不得同时包含其他工具调用。

### FR-01 AgentRunner 工具驱动循环

- 系统必须提供 AgentRunner，支持多轮 function calling。
- AgentRunner 每轮调用 LLM 后，必须按 OpenAI tool calling 格式追加结构化 messages。
- 如果响应包含 tool calls，系统必须执行对应工具，并将结果作为 `role=tool` 消息写回。
- 如果响应不包含 tool calls，循环结束。
- 每个决策周期最后必须通过 `ctrl.commit(staging_hash)` 或 `ctrl.abort()` 结束。
- `ctrl.commit()` 后 Agent 进入休眠，由 Timer Scheduler 接管。

### FR-02 System Prompt

- System Prompt 必须由四部分组成：身份与运行时、当前局势、可用工具摘要、可用技能摘要。
- 当前局势必须由 `game_state.json` 生成 `game_state.md` 后注入。
- 工具部分只放命名空间摘要，不展开全部参数。
- 技能部分必须列出可用 skill，支持 always skill 自动注入和非 always skill 按需加载。
- System Prompt 不应包含旧架构的多角色指令、强制 8 步流程或 YAML 输出模板。

### FR-03 工具注册与发现

- 系统必须提供 ToolRegistry 注册所有工具。
- 工具名称必须采用 `namespace.action` 形式。
- 工具必须声明 JSON Schema 参数、描述、只读/写入属性和执行 handler。
- Agent 可通过 `ctrl.discover_tools(namespace)` 查询命名空间下的工具详情。
- 执行类工具必须返回成功/失败和原因。

### FR-04 观测与查询工具

- `obs.*` 必须读取当前帧游戏状态，不修改游戏状态。
- `query.*` 必须基于当前状态提供条件检索。
- 所有单位 tag、建筑 tag 和敌方可见信息必须来自 `obs.*` 或 `query.*`，禁止 Agent 编造。

### FR-05 指令、建造、经济与编队工具

- `cmd.*` 必须映射 burnysc2 原生命令。
- `build.*` 必须覆盖建筑、训练、研发、挂件、取消、起飞、降落等生产动作。
- `econ.*` 必须覆盖采集、转移工人、气矿分配、扩张、建气矿。
- `squad.*` 必须覆盖小队创建、增删成员、解散、自动均分和批量命令。

### FR-06 Timer Scheduler

- 系统必须提供 `timer.command(at_time, tool_name, arguments)`，按游戏时间执行结构化工具调用。
- `timer.command` 禁止使用 `call="build.train(...)"` 字符串形式作为执行 API；字符串只能用于日志展示。
- 系统必须提供 `timer.monitor(metric, op, value, ...)`，在条件满足时唤醒 Agent。
- 时间唤醒也必须使用 `timer.monitor(metric="game_time")`，不单独提供 `timer.schedule`。
- monitor 必须支持 `before_time`，超时后自动注销。
- 唤醒 Agent 时必须附带 reason，作为本轮 user message 的上下文。
- Timer Scheduler 必须每帧执行到期命令并评估活跃 monitor。

### FR-07 时间线规划

- 主 Agent 必须自己完成时间线规划，不使用规划 subAgent。
- `plan.simulate(commands, horizon)` 必须作为确定性时间线模拟器，纳入 active timers、当前生产队列、补给占用、建筑占用和工人占用。
- `plan.build_time` 必须提供单位、建筑、挂件和升级的耗时与成本。
- `plan.build_order` 必须提供标准开局模板目标列表。
- 所有 `at_time` 命令必须经过 `plan.build_time` 和 `plan.simulate` 辅助推算。
- `plan.simulate` 必须返回每步预测资源、预测补给、关键假设和第一个失败点。
- 命令总消耗不得超过预测资源收入。

### FR-08 审查工具

- 提交前必须调用 `review.plan` 审查最终命令列表。
- `review.plan` 必须绑定当前 `staging_hash`；`ctrl.commit(staging_hash)` 只允许提交同一个 hash。
- `review.plan` 必须包含 `review.params` 与 `review.logic`。
- `review.params` 必须纯代码检查 tag、坐标、放置位置、生产建筑、科技前置、资源和冲突。
- `review.logic` 必须 spawn 审查 Sub-Agent，审查对抗合理性、资源合理性、战略一致性和时间线合理性。
- 审查结果必须包含 `verdict: PASS | WARN | REVISE`、issues 和 suggestions。
- `WARN` 不应强制阻止提交，主 Agent 拥有最终决策权。

### FR-09 Sub-Agent

- Sub-Agent 必须由主 Agent 通过工具 handler 同步调用。
- Sub-Agent messages 必须与主 Agent 隔离。
- Sub-Agent ToolRegistry 必须最小化，只暴露任务所需工具。
- Sub-Agent 必须禁用 `ctrl.*`、`timer.*` 和递归 spawn。
- Sub-Agent 失败时 handler 不应抛异常，而应返回部分进度和失败原因。

### FR-10 记忆系统

- 系统必须将每次 wake-up 周期的完整 messages 追加写入 JSONL 文件。
- 原始 messages 不得删除或重写。
- `last_consolidated` 必须标记已经整合进 `game_state.json` 的消息位置。
- Agent 下次唤醒上下文必须由 `game_state.md`、游标之后的最近 raw messages 和本轮 user message 组成。
- `get_history()` 裁剪历史时必须参考 nanobot 的 `_find_legal_start()` 思路，保证不会产生 orphan tool result。
- `hist.*` 工具必须能查询旧 messages、事件和快照。

### FR-11 game_state.json 与 game_state.md

- `game_state.json` 是权威源，只由代码和 MemoryConsolidator 写入。
- `game_state.md` 是派生视图，只用于注入 System Prompt 和给 LLM 阅读。
- `game_state.json` / `game_state.md` 必须包含四段：战略判断、当前优先级、已知事实、关键事件。
- 每段必须标注更新时间和游戏时间。
- 已知事实和关键事件由代码每周期确定性维护。
- 战略判断和当前优先级由整合 LLM 在 token 超预算时维护。
- Agent 必须能通过更新时间判断信息是否过时。

### FR-12 快照与事件日志

- 系统必须维护三层 botAI 快照：
  - 分钟级：每 60 秒，无上限。
  - 决策级：每次 `ctrl.commit()`，保留最近 5 份。
  - 秒级：每 5 秒，保留最近 5 份。
- 系统必须维护事件日志 JSONL。
- `hist.snapshot`、`hist.trend`、`hist.events`、`hist.unit`、`hist.compare` 必须基于这些数据工作。

### FR-13 MemoryConsolidator

- MemoryConsolidator 必须在每次 commit 后执行代码更新。
- 当游标之后消息 token 超预算时，必须触发整合 LLM。
- 整合 LLM 输入应包含旧战略判断、旧优先级和事实差异摘要。
- 整合 LLM 只能更新战略判断和当前优先级。
- 整合完成后必须推进 `last_consolidated`。
- 在 stop-the-world 模型下，MemoryConsolidator 必须在 commit 后、sleep 前同步执行完毕，不做后台异步写入。

### FR-14 技能系统

- 技能目录必须支持 `skills/{name}/SKILL.md`。
- Skill 文件使用 YAML frontmatter + Markdown 正文。
- `always: true` 的 skill 必须自动注入。
- 非 always skill 必须通过 `skill.load(name)` 按需加载。
- workspace skill 必须优先于 builtin skill。
- 初始设计建议的技能包括：`main-flow`、`production-math`、`timeline-planning`、`standard-openings`、`review-knowledge`、`review-dimensions`、`consolidation-guide`、`monitor-calibration`。

### FR-15 旧架构清理

- 系统必须移除 StrategyLayer、PhaseLayer、ExecutionLayer。
- 系统必须移除 Supervisor 主流程。
- 系统必须移除旧 prompt 文件。
- 系统必须移除旧 TriggerScheduler。
- 可保留 LLMClient、ProductionManager、SquadManager、TacticManager、observation、DecisionLogger、strategy_pool 等基础设施，但需要重新挂接到新架构。

## 7. 非功能需求

### 7.1 可靠性

- 工具 handler 不应让异常直接击穿 AgentRunner。
- 定时命令失败必须记录失败原因，并可通过历史工具查询。
- Sub-Agent 失败必须返回结构化失败结果。
- monitor 必须支持自动过期，避免长期误触发。

### 7.2 性能

- 常规工具调用应保持轻量。
- `review.params` 目标执行时间约 1ms。
- 快照总量在 20 分钟对局中约数十份，目标存储量低于 1MB。
- MemoryConsolidator 使用便宜 LLM，目标延迟 1-2 秒，在 commit 后、sleep 前同步执行。

### 7.3 可维护性

- 工具必须原子化，避免把复杂策略封装进高级 wrapper。
- 命名空间必须稳定，便于 Agent 通过名称理解能力边界。
- 领域知识应沉入 skill，而不是堆入 System Prompt。
- 消息与记忆文件采用追加或分段更新，降低解析和回滚复杂度。

### 7.4 可观测性

- 每次 wake-up、tool call、commit、abort、timer command 执行、monitor 触发、整合触发都必须记录。
- 日志必须包含 game_time、wake_id、tool_name、参数摘要、结果摘要、错误码。
- 应能通过历史工具复盘一次决策周期。

### 7.4.1 FR-16：对局复盘可视化

系统必须提供一个单文件 HTML 页面，用于加载并可视化 `storage/{game_id}/` 中的运行时数据。

**数据源**：
- `logs/app.log` — JSON 行格式的应用日志
- `sessions/default.jsonl` — Agent 完整 messages 账本
- `memory/game_state.md` — 世界模型派生视图

**页面结构**（三个面板）：

**面板一：时间线总览**

横轴为游戏时间，纵轴按事件类型分层：
- Agent 周期（色块：绿=committed，黄=aborted，红=error），标注 wake_id
- LLM 调用（竖线，粗细代表延迟，悬停显示 token 数 / 模型 / 耗时）
- 工具调用（点标注，颜色按命名空间分类）
- 游戏事件（timer 触发、monitor 唤醒）

点击任意节点展开详情。

**面板二：资源与状态曲线**

折线图，三条线：minerals、vespene、supply_used，数据来源为日志中 `event="game.state"` 的记录。下方标注 Agent 决策点（虚线标记 commit/abort 时刻）。

**面板三：事件详情列表**

可过滤、可搜索的表格。列：game_time、level、event、message、关键字段。过滤条件：按 wake_id、按 event 类型、按 LEVEL。

**技术要求**：
- 单文件 HTML，零服务端依赖，约 800 行以内
- 使用 Chart.js CDN（折线图）或纯 Canvas 自绘
- 用户拖拽 `app.log` 即可加载（FileReader API），或指定目录路径
- JSON 行格式天然适配，无需修改日志格式

### 7.5 安全性

- Agent 不得编造 unit_tag。
- 任何影响游戏状态的能力必须通过工具 schema 和 handler 校验。
- `exec.*` 如后续实现，必须只读、限时、限内存、禁文件 I/O、禁网络、输出截断。

## 9. 验收标准

- 启动一局 SC2 后，Agent 首轮上下文只包含新 System Prompt、`game_state.md`、工具摘要和唤醒原因。
- Agent 能通过 `obs.*` 与 `query.*` 获取必要信息，而不是接收全量观测 dump。
- Agent 能生成不少于 60 秒的稳定开局时间线，并注册 `timer.command` 与 `timer.monitor`。
- Timer Scheduler 能按游戏时间执行命令，并在 monitor 条件满足时提前唤醒 Agent。
- 提交前可观察到 `review.plan` 调用，且 `review.params` 与 `review.logic` 分别执行。
- commit 后 Session JSONL、events、快照和 `game_state.md` 均被更新。
- token 超预算时，MemoryConsolidator 能推进 `last_consolidated` 并更新战略判断和当前优先级。
- 高频唤醒时，Agent 能通过 `hist.events` 识别是否需要调整 monitor 条件。
- 旧 Strategy/Phase/Executor/Supervisor 主路径不再参与决策。

## 10. 实施优先级

| 优先级 | 内容 |
|--------|------|
| P0 | AgentRunner、ToolRegistry、`obs.*`、`query.*`、`timer.*`、`ctrl.*`、基础 Session |
| P1 | `cmd.*`、`build.*`、`plan.*`、Timer Scheduler、`review.params` |
| P2 | `review.logic` Sub-Agent、MemoryStore、RoundRecorder、快照、`hist.*` |
| P3 | MemoryConsolidator、SkillLoader、System Prompt 完整集成 |
| P4 | 旧架构清理、实战调参、monitor 校准、差异分析 Sub-Agent |
| P5 | `exec.*` 沙盒执行器 |
